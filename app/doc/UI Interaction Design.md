### 1. 全局设计规范 (Global Design System)
对于低视力用户，视觉清晰度是第一位的；对于全盲用户，焦点导航逻辑是第一位的。

+ **配色方案 (Color Palette)**：
    - **背景**：纯黑 (`#000000`) —— 最大程度减少眩光，省电。
    - **主色**：高亮黄 (`#FFFF00`) —— 视觉最敏感的颜色，用于核心按钮。
    - **辅色**：纯白 (`#FFFFFF`) —— 用于辅助文本。
    - **警告色**：亮橙 (`#FF5722`) —— 用于障碍物警告。
+ **排版 (Typography)**：
    - **字体**：Roboto 或系统默认无衬线字体，加粗 (Bold)。
    - **字号**：最小文本不低于 `24sp`，标题 `32sp`<sup>2</sup>。
+ **触控目标 (Touch Targets)**：
    - 最小点击区域：`60dp x 60dp`（远超安卓建议的 48dp）。

### 2. 核心页面交互设计
#### 2.1 首页：双屏控制中心 (Home Screen)
为了降低认知负荷，首页不使用底部的 TabBar，而是采用**上下分屏的大卡片布局**。视障用户手指一划就能找到。

+ **布局结构**：
    - **上半屏 (50%) - 导航模式**：背景黑色，中间显示巨大的黄色“导航”图标。
    - **下半屏 (40%) - 视觉/识字模式**：背景深灰色，中间显示巨大的白色“识字”图标。
    - **底部 (10%) - 语音/设置条**：长条形按钮，用于唤醒帮助或进入设置。
+ **无障碍焦点顺序 (TalkBack Flow)**：
    1. 聚焦上半屏 -> 读出：“导航模式。双击进入。”
    2. 聚焦下半屏 -> 读出：“文字识别模式。双击进入。”
    3. 聚焦底部 -> 读出：“语音助手与设置。”

#### 2.2 导航模式页面 (Navigation UI)
进入导航后，屏幕视觉元素必须极简，避免信息过载。

+ **待机状态 (输入目的地)**：
    - **全屏按钮**：屏幕中央一个巨大的麦克风图标。
    - **交互**：单击屏幕任意位置 -> 提示“请说话” -> 用户说“去食堂” -> 系统确认。
+ **行进状态 (导航中)**：
    - **顶部 (20%)**：显示下一指令（如“左转”），配合巨大的转向箭头（占满宽度）。
    - **中部 (60%)**：显示当前路况/障碍物提示。平时显示“路况良好”，遇到障碍时变红并显示“前方障碍”。
    - **底部 (20%)**：显示剩余距离和时间。
+ **冲突交互 (避障优先)**：
    - 当检测到障碍物时，屏幕会闪烁（针对低视力），同时弹出模态窗口（Modal）覆盖当前导航信息，强制读屏软件读取“前方障碍” <sup>3</sup>。

#### 2.3 文字识别页面 (OCR UI)
+ **取景器**：
    - 全屏显示摄像头画面。
    - **自动闪光灯**：当光线传感器检测到暗光时，自动开启闪光灯，并在屏幕上显示“已开启补光”提示图标 <sup>4</sup>。
+ **交互逻辑**：
    - **实时模式（默认）**：用户移动手机，只要 ML Kit 识别到文字，TalkBack 直接朗读。
    - **锁定/暂停**：双击屏幕任意位置，画面冻结（Pause），方便用户仔细听长段文字。再次双击恢复扫描。

### 3. 听觉与触觉交互规范 (Audio & Haptic UX)
这是 EarRove 的核心体验，必须建立一套标准化的“语言”。

#### 3.1 震动语言 (Haptic Language)
根据 PRD v1.0 的定义：

| **场景** | **震动模式** | **频率模拟** | **含义** |
| --- | --- | --- | --- |
| **可交互控件** | 极短震动 | `Tick`<br/> (5ms) | 手指摸到了一个按钮（边界感知） |
| **障碍物警报** | 短震两下 | `Buzz-Buzz`<br/> (100ms, 间隔50ms) | 前方危险，注意避让 |
| **转向提示** | 长震一下 | `Hummmm`<br/> (400ms) | 到达路口，准备转向 |
| **操作确认** | 中震一下 | `Click`<br/> (50ms) | 语音输入成功/模式切换成功 |


#### 3.2 语音反馈 (Audio Feedback)
+ **Earcon (提示音)**：在 TTS 说话前使用简短的音效。
    - _叮_ (清脆高音) = 开始听写。
    - _咚_ (低沉音) = 识别失败/网络错误。
+ **TTS 优先级**：
    - **Level 1 (最高)**：障碍物警报、电量过低关机。 -> **行为：打断所有声音立即播放**。
    - **Level 2 (中等)**：导航转向指令、红绿灯倒计时。 -> **行为：排队播放**。
    - **Level 3 (最低)**：环境文字朗读、无关提示。 -> **行为：如果 Level 1/2 正在播放，则丢弃不播**。

### 4. 这里的坑：Android TalkBack 适配细节
在开发阶段（UI 编写）时，你需要特别注意以下几点技术实现，否则设计无法落地：

1. **ContentDescription 是灵魂**：
    - 对于 ImageView（如箭头、图标），必须设置 `android:contentDescription="左转箭头"`。
    - 对于装饰性图片（如分割线），必须设置 `android:contentDescription="@null"`，否则读屏软件会读出“未加标签的按钮”，让盲人困惑 <sup>5</sup>。
2. **焦点顺序 (Traversal Order)**：
    - XML 布局中默认是从上到下、从左到右。
    - 如果你的布局很复杂，需要使用 `android:accessibilityTraversalBefore` 或 `After` 来强制规定朗读顺序。
3. **触摸区域扩大**：
    - 虽然 UI 上图标可能是 40dp，但在代码中请使用 `TouchDelegate` 或给父容器设置点击事件，确保用户点在图标旁边 1cm 处也能响应。
